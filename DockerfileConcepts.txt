# 🧩 Step 1: What Docker actually does

When you run:

```bash
docker run my-app
```

👉 Docker does **NOT create a VM**
👉 It creates a **process on the host OS**

BUT it modifies that process using kernel features so it *looks like a separate machine*

---

# 🔐 Step 2: Namespaces → Isolation

## 🧠 Core idea

👉 Namespaces give a process its **own view of the system**

---

## 💡 Without namespaces

A process can see:

* all processes
* all files
* all network interfaces

👉 No isolation

---

## 💡 With namespaces

Docker tells kernel:

👉 “Create a process that sees its own world”

---

## 🔍 What gets isolated

### 1. PID Namespace (Processes)

* Container sees:

  ```text
  PID 1 → its own app
  ```

* But host may have:

  ```text
  PID 4567 → same process
  ```

👉 Container thinks it is the only process

---

### 2. NET Namespace (Networking)

Each container gets:

* its own IP
* its own ports
* its own network stack

👉 That’s why:

* multiple containers can use same port internally

---

### 3. MOUNT Namespace (Filesystem)

* Container sees only:

  * its own filesystem (image layers)

👉 Cannot see host files unless mounted

---

### 4. UTS Namespace (Hostname)

* Each container has its own hostname

---

### 5. IPC Namespace

* Isolates communication between processes

---

## 🎯 Summary

👉 Namespaces = **isolation layer**

---

# ⚙️ Step 3: cgroups → Resource Control

## 🧠 Core idea

👉 cgroups control **how much resources a container can use**

---

## 💡 Without cgroups

One container can:

* consume all memory
* use all CPU
* crash system

---

## 💡 With cgroups

Docker tells kernel:

👉 “Limit this process”

---

## 🔍 What can be controlled

* CPU usage
* Memory usage
* Disk I/O
* Network bandwidth

---

## 🧪 Example

```bash
docker run -m 512m --cpus=1 my-app
```

👉 Means:

* max 512MB RAM
* max 1 CPU

---

## 🎯 Summary

👉 cgroups = **resource limits + monitoring**

---

# 🧩 Step 4: Combine Both (THIS IS THE MAGIC)

When Docker creates a container:

---

## 🔹 Step 1: Create process

* normal Linux process

---

## 🔹 Step 2: Apply namespaces

👉 isolate:

* processes
* network
* filesystem

---

## 🔹 Step 3: Apply cgroups

👉 limit:

* CPU
* memory

---

## 🔹 Step 4: Attach filesystem (image layers)

👉 container gets its own environment

---

# 🧠 Final Mental Model

👉 A container is:

> **A process + namespaces (isolation) + cgroups (limits)**

# 🔥 Why this makes Docker powerful

Because:

* No full OS needed

* No guest kernel

👉 Uses host kernel directly

# ⚡ Why containers are lightweight

* share kernel

# 🐳 Docker Base Images – Complete Reference Notes

---

# 1. What is a Docker Base Image?

A Docker base image is a **prebuilt filesystem snapshot** that contains:

* system libraries
* binaries
* runtime environments (Java, Python, etc.)
* minimal OS userspace

It is used as the starting point in a Dockerfile.

---

## Example

```dockerfile
FROM ubuntu
FROM alpine
FROM eclipse-temurin:17-jre
```

---

## Key Idea

Docker images contain:

* userspace (tools, libraries, binaries)

Docker images do NOT contain:

* kernel ❌

---

# 2. Docker Image vs OS vs Kernel

---

## OS = Kernel + Userspace

```text
OS = Kernel + Libraries + Tools + Shell + Services
```

---

## Docker Image = Userspace only

```text
Docker Image = Libraries + Tools + Filesystem (NO kernel)
```

---

## Container = Userspace + Host Kernel

```text
Container = Image userspace + shared host kernel
```

---

## Key Insight

👉 All containers share the **same kernel (host OS kernel)**
👉 Images differ only in **userspace**

---

# 3. What Kernel do Docker Images use?

👉 **None of the images have their own kernel**

All of these:

```dockerfile
FROM maven:3.9.6-eclipse-temurin
FROM eclipse-temurin:17-jdk
FROM eclipse-temurin:17-jre
FROM eclipse-temurin:17-jre-alpine
```

👉 use the **same host kernel**

---

## On macOS

* Docker runs inside a Linux VM
* That VM provides the kernel
* All containers share that kernel

---

# 4. What is Linux Userspace?

Userspace = everything except kernel.

It includes:

* C library (glibc / musl)
* shell (bash / sh)
* package manager (apt / apk / yum)
* system utilities (ls, cp, etc.)
* filesystem layout
* system tools/services

---

# 5. Types of Linux Userspaces (Distros)

---

## 1. Debian / Ubuntu (glibc-based)

Characteristics:

* uses glibc
* package manager: apt
* full-featured
* large size

Example:

```dockerfile
FROM ubuntu
```

---

## 2. RedHat / CentOS (glibc-based)

Characteristics:

* uses glibc
* package manager: yum / dnf
* enterprise-focused

---

## 3. Alpine (musl-based)

Characteristics:

* uses musl libc
* package manager: apk
* very small (~5MB)

Example:

```dockerfile
FROM alpine
```

---

## 4. Distroless / Minimal

Characteristics:

* no shell
* no package manager
* only runtime

Example:

```dockerfile
FROM gcr.io/distroless/java17
```

---

# 6. C Library (libc)

---

## What is libc?

A C library provides system-level functions:

* file operations
* memory management
* networking
* process control

---

## Flow

```text
App → libc → kernel → hardware
```

---

## Types

### glibc

* used in Ubuntu/Debian
* large, feature-rich
* highly compatible

---

### musl

* used in Alpine
* lightweight
* simpler
* sometimes less compatible

---

## Key Difference

| Feature       | glibc  | musl   |
| ------------- | ------ | ------ |
| Size          | Large  | Small  |
| Compatibility | High   | Lower  |
| Used in       | Ubuntu | Alpine |

---

# 7. Why Alpine is different

---

## Uses BusyBox

Instead of full GNU tools, Alpine uses BusyBox:

* single binary for many tools
* minimal utilities

---

## Uses `sh`, not `bash`

Reason:

* bash is large
* Alpine focuses on minimalism

---

## Install bash (optional)

```bash
apk add bash
```

---

## Key Insight

Alpine = minimal → smaller images → faster startup
But → more compatibility issues

---

# 8. Why Java sometimes breaks on Alpine

---

## Root Cause

* Java expects glibc
* Alpine uses musl

---

## Problems

* native libraries fail
* JNI issues
* threading differences
* DNS resolution differences

---

## Example Errors

```text
java.lang.UnsatisfiedLinkError
```

---

## Solution

Use glibc-based image:

```dockerfile
FROM eclipse-temurin:17-jre
```

---

# 🎯 Final Interview Answer

> Docker base images provide different userspace environments on top of the same host kernel. They differ in system libraries like glibc or musl, available tools, package managers, and overall size. For example, Ubuntu-based images use glibc and are more compatible, while Alpine uses musl and is smaller but may cause compatibility issues. Java images like eclipse-temurin provide a production-ready OpenJDK distribution. The choice of base image depends on trade-offs between size, compatibility, and security.
